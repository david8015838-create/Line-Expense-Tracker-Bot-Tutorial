00什麼是 n8n？

先花兩分鐘搞懂這個工具，後面所有節點設定才會有概念。

n8n 是一套開源的「工作流自動化」工具。你可以把它想像成 樂高積木：每一塊積木（稱為「節點 Node」）負責一件小事， 例如「接收一個 LINE 訊息」「呼叫 AI 幫忙分析」「在 Google Sheets 新增一列」「傳訊息回 LINE」。 把這些積木用線連起來，資料就會像水流一樣，從第一個節點依序流到最後一個節點，全程不用自己寫後端伺服器。

🧩
視覺化節點
每個節點是一個方框，代表一個動作。點兩下打開就能設定參數，不用寫程式（需要彈性時也可以加 Code 節點寫 JavaScript）。
🔌
串接各種服務
內建上百種服務的整合（Google Sheets、Gmail、Slack、LINE、OpenAI、Gemini...），不用自己研究每家的 API 文件。
💻
自架 = 免費
n8n 可以自己架在自己的主機上（Docker 一行指令就能跑），完全免費、資料自己掌控；也有付費雲端版，但本教學用自架版即可。
🔄
資料怎麼流動
每個節點接收上一個節點的輸出（JSON 格式），處理後再輸出給下一個節點，整條 workflow 像是一條輸送帶。

這個 Bot 用 n8n 做什麼？

簡單說，整個記帳 Bot 就是一條 n8n workflow：用戶在 LINE 傳「午餐120」或拍一張發票照片 → n8n 收到後讓 Gemini AI 判斷這是「記帳」還是「查詢消費」 → 記帳就寫進 Google Sheets， 查詢就讀取 Sheets 算出統計結果 → 最後把結果用 LINE 傳回給用戶。

💡 為什麼選 n8n 而不是自己寫程式？

自己寫一個 LINE Bot 後端，需要架伺服器、處理 webhook、寫資料庫存取、部署上線，至少要懂一種後端語言。 n8n 把這些都包成「節點」，你只要拖拉設定，搭配少量 Code 節點補足彈性， 幾個小時就能做出一個完整、可上線的 Bot——這正是教學給同事的最大優勢：門檻低、可視化、好維護。

0-4　LINE Bot 設定

建立 LINE Official Account，並開啟 Messaging API 功能。

📖 名詞對照：先搞懂這幾個詞的關係
Provider
最上層的「開發者帳號」分類，一個 Provider 底下可以有多個 Official Account / Channel。先建立一個 Provider 當容器即可。
LINE Official Account（OA）
「官方帳號」本身——使用者在 LINE 上加好友、聊天的對象，之後用戶都是跟這個帳號互動。
Messaging API Channel
把一個 Official Account「升級」成可以被程式控制的頻道，啟用後才能用 API 收發訊息。一個 OA 對應一個 Messaging API Channel。
Channel secret
用來驗證 Webhook 來源是否為 LINE 官方的密鑰（本教學流程中不會直接用到，但要知道它和 Token 不是同一個東西）。
Channel access token
呼叫 LINE API（例如 Push Message 主動傳訊息給用戶）時要附帶的「通行證」，是本教學最重要、會實際用到的憑證。
⚠️ 兩個重要限制

LINE 已不支援從 Developer Console 直接建立 Messaging API channel，需先建立 LINE Official Account， 再從官方帳號後台開啟 Messaging API。另外，帳號名稱不能包含「LINE」字樣，這個詞被官方保留使用。

這個設計為什麼長這樣？

  • 寫入 Google Sheet 用「直接呼叫 API」的方式：這樣可以自由決定要寫進哪一欄、哪一列，比較有彈性。
  • 呼叫 LINE 和 Gemini 都用「程式碼節點」自己組訊息：這樣可以完全控制傳出去的內容長什麼樣子。
  • 多了一個「加入日期」節點：讓 AI 先知道「今天是幾號」，使用者問「昨天」「這個月」花多少錢時才能算得準。
  • LINE 訊息一進來就先「立刻回應」，結果之後再補發一則訊息：因為 AI 處理需要幾秒鐘，先回應可以避免 LINE 那邊顯示逾時或失敗，體驗比較順暢。
  • AI 回傳的內容會先經過一道「清理」程序：AI 偶爾會多加一些格式符號，先清乾淨，後面的步驟才能正確讀懂。
  • 收到圖片訊息時，會多一個步驟去下載圖片：因為 LINE 一開始只會給「這張圖片的編號」，要再跟 LINE 要一次圖片本身才看得到內容。

LINE 要求 Webhook 在 1 秒內回傳 200 OK，否則會重試發送。 但 AI 處理（辨識文字/圖片、查詢統計）通常需要 3-10 秒，如果用「Last node」回應會超時報錯， 而且 LINE 的 Reply Token 只能用一次——選 Immediately 可以先回 200 OK 給 LINE， 後續節點繼續在背景跑，最後改用 Push Message API 主動把結果傳給用戶。

💡 為什麼 rows 要設計成「扁平陣列」？

Google Sheets 新增資料列時，需要的格式就是「一列一個陣列」，例如 [日期, 時間, 分類, 描述, 金額]。把資料整理成這種扁平陣列， 後面「寫入 Sheet」節點就能直接拿來用，不用再多一層轉換。

💡 為什麼多筆消費時，每一行都要顯示自己的日期？

如果使用者一次輸入好幾筆不同天的消費（例如「6/1出去玩1200、6/2買手機30000」）， 每一筆其實是不同日期的紀錄。所以回覆訊息在多筆時，會讓每一行都顯示自己的 MM/DD， 最後再加總總金額，這樣使用者一眼就能看出每一筆分別是哪天花的。

💡 為什麼分類用「一個表情符號」表示，而不是文字？

LINE 的訊息框寬度有限，如果分類用文字（例如「旅遊住宿」）再加上金額，整行會太長， 在手機上容易被截斷換行，金額看起來就會跑到下一行、整體很亂。 改用單一表情符號代表分類（例如 ✈️ 代表旅遊住宿），可以讓每一行大幅縮短， 大部分情況下都能在一行內顯示完整，閱讀起來更整齊。

📌 為什麼先把 parsed 統一包成陣列？

JSON.parse($('If').item.json.output) 可能是「單一物件」（一筆消費）， 也可能是「陣列」（使用者一次輸入多筆消費時）。 用 items = Array.isArray(parsed) ? parsed : [parsed] 統一轉成陣列後， 後面的程式碼（算 rows、算合計、組 reply）就只需要寫一套， 不用為「單筆」和「多筆」各寫一份。

依 AI 給的日期區間篩選紀錄，按分類統計加總，組成漂亮的分析報告文字。

JavaScript — 處理查詢 
const ai = JSON.parse($('If').item.json.output);
const records = $input.all().map(i => i.json);
const userId = $('Webhook').item.json.body.events[0].source.userId;

const start = ai.start_date;
const end = ai.end_date;
const filtered = records.filter(r => r.date >= start && r.date <= end);

if (filtered.length === 0) {
  const s = start.slice(5).replace('-','/');
  const e = end.slice(5).replace('-','/');
  return [{ json: { reply: `📭 ${s}～${e} 沒有消費紀錄`, userId } }];
}

const byCategory = {};
filtered.forEach(r => {
  byCategory[r.category] = (byCategory[r.category] || 0) + Number(r.amount);
});

const total = filtered.reduce((sum, r) => sum + Number(r.amount), 0);

const emoji = {
  '餐飲':'🍱','飲料咖啡':'☕','大眾交通':'🚌','油資':'⛽',
  '停車費':'🅿️','計程車Uber':'🚕','購物雜貨':'🛒','服飾':'👗',
  '租金':'🏠','水電瓦斯':'💡','電話網路費':'📱','3C電器':'💻',
  '訂閱服務':'📺','醫療藥品':'💊','美容美髮':'💇','健身運動':'🏃',
  '娛樂':'🎮','旅遊住宿':'✈️','教育學習':'📚','書籍文具':'📖',
  '保險':'🛡️','投資理財':'📈','孝親費':'👨‍👩‍👧','禮金紅包':'🧧',
  '寵物':'🐾','其他':'📦'
};

const lines = Object.entries(byCategory)
  .sort((a, b) => b[1] - a[1])
  .map(([cat, amt]) => {
    const pct = Math.round(amt / total * 100);
    const icon = emoji[cat] || '📦';
    return `${icon} ${cat}　$${Number(amt).toLocaleString()}　${pct}%`;
  });

const s = start.slice(5).replace('-','/');
const e = end.slice(5).replace('-','/');

const reply = [
  `📊 消費分析（${s}～${e}）`,
  '───────────────────',
  ...lines,
  '───────────────────',
  `💰 合計  $${Number(total).toLocaleString()}｜共 ${filtered.length} 筆`
].join('\n');

return [{ json: { reply, userId } }];

⚠️ 為什麼不用 Append Row 節點？

n8n v1.123.5 的 Google Sheets Append Row 節點，欄位值的表達式（{{ $json.date }} 等） 無法正確解析，會把表達式原文寫入 Sheet，而不是實際值。改用 HTTP Request 直接呼叫 Google Sheets API， Body 用物件表達式語法 {{ {...} }} 即可正確序列化。

💡 為什麼 Body 要用 {{ {"values": $json.rows} }} 而不是字串？

這個表達式回傳的是一個 JavaScript 物件，n8n 會自動正確序列化成 JSON。 如果改成手寫字串模板（例如 {"values": [["{{ $json.date }}", ...]]}）， 遇到含特殊字元或換行的欄位值就容易產生「JSON parameter needs to be valid JSON」錯誤。

⚠️ 為什麼用 Code 節點而不是 HTTP Request 節點？

n8n 的 HTTP Request 節點在 JSON body 模板中，無法正確處理含 \n 換行字元的表達式 （reply 內容含換行符），會出現「JSON parameter needs to be valid JSON」錯誤。 改用 Code 節點的 this.helpers.httpRequest()，讓 JavaScript 原生處理 JSON 序列化，問題就解決了。
